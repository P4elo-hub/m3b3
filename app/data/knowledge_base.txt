- DocIntel: AI-ассистент для извлечения, генерации и поиска по проектной документации на основе Markdown-файлов и AI-моделей
- Назначение: централизованное взаимодействие с технической документацией системного аналитика
- Импорт из Confluence: выгрузка пространств через REST API, конвертация страниц в единый Markdown с сохранением заголовков, таблиц и вложений
- Единый формат документов: Markdown с YAML front matter (title, author, version, tags, source)
- Хранение: git-репозиторий документации с версионированием и историей изменений
- Семантический поиск: инструмент search_kb ищет релевантные фрагменты по ключевым словам и контексту запроса
- Создание документов: через интерфейс DocIntel или из шаблонов — BRD, FRD, API-spec, ADR (Architecture Decision Record)
- Редактирование: изменения сохраняются в Markdown-файлах, доступен предпросмотр и diff перед коммитом
- Шаблон BRD: разделы «Цели», «Заинтересованные стороны», «Бизнес-требования», «Ограничения», «Критерии приёмки»
- Шаблон FRD: разделы «Функциональные требования», «Нефункциональные требования», «Диаграммы», «Зависимости»
- Шаблон API-spec: OpenAPI 3.0, для каждого endpoint — метод, путь, параметры, коды ответов, примеры
- Глоссарий: BRD — Business Requirements Document, FRD — Functional Requirements Document, ADR — Architecture Decision Record, SA — System Analyst
- Проект «Платёжный шлюз» FR-001: авторизация платежа должна завершаться не дольше 3 секунд при нагрузке до 500 TPS
- Проект «Платёжный шлюз» FR-002: поддержка интеграции с СБП (Система быстрых платежей) и картами МИР
- Проект «Платёжный шлюз» NFR-001: доступность сервиса 99.9%, RPO 15 минут, RTO 1 час
- API payments-api POST /v1/payments: создание платежа; обязательные поля amount (decimal), currency (ISO 4217), merchant_id (uuid)
- API payments-api GET /v1/payments/{id}: статус платежа; возможные значения status: pending, authorized, failed, refunded
- ADR-007 Платёжный шлюз: выбран event-driven подход с Kafka для асинхронной обработки уведомлений от банков-эквайеров
- Регламент документирования: каждое функциональное требование имеет уникальный ID (FR-NNN), версию и автора в front matter
- Экспорт в Confluence: обратная синхронизация Markdown → Confluence доступна для пространств с меткой docintel-sync
